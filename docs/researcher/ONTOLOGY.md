# Ontology Reference

## Hierarchy

Regime -> market state -> context -> event -> canonical family -> template -> hypothesis

## Layer definitions

### Regime
A coarse market condition used to stratify evaluation and discovery.

### Market state
A derived state label that provides operational context for signals.

### Context
A set of conditions that narrow a hypothesis.

### Event
A detectable market episode or trigger.

### Canonical family
The stable ontology boundary that determines template eligibility and directional semantics.

### Template
The structural pattern used to express the trade idea.

### Hypothesis
The fully specified unit that is evaluated, stored, and promoted.

## Contract rules

- The canonical family list is stable unless the ontology itself is updated.
- Hypotheses must be fully specified.
- Direction semantics must remain compatible with family membership.
- The docs should not introduce casual "misc" categories that bypass the contract.

## Where the ontology is used

- detector and event registries,
- hypothesis generation,
- memory deduplication,
- multiplicity correction,
- and promotion gating.

## What to keep in mind

The ontology is not just naming. It is a constraint system.
